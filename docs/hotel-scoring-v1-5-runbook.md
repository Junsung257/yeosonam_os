# 호텔 점수 V1.5 운영 런북

> 목적: 같은 5성이라도 호텔 체급이 다른 문제를 고객 추천/비교에 반영한다. 고객 화면에는 숫자 점수를 숨기고, 운영 화면에는 근거와 누락률을 보여준다.

## 1. 고객 표시 원칙

- 고객에게 `97점` 같은 절대 점수는 보여주지 않는다.
- 고객에게는 `호텔 우수`, `호텔 무난`, `호텔 확인 필요` 세 단계만 보여준다.
- 호텔 근거는 짧게 쓴다.
  - `호텔 조건이 좋은 편이에요`
  - `호텔 조건은 무난해요`
  - `호텔 등급 정보가 부족해요`
- 리뷰가 없어도 `package_scores` 비교군이 있으면 “여소남 비교판정”으로 노출한다.

## 2. V1.5 점수 입력값

현재 구현 기준:

- `hotel_avg_grade`: 일정/상품에서 추출한 호텔 평균 등급
- `breakdown.mrt_hotel_quality_score`: MRT 호텔 인텔 또는 외부 호텔 품질 점수
- `mrtMatchScore`: 호텔명 매칭 신뢰도, 별도 연결 시 사용
- `brandWithinStarScore`: 같은 성급 안에서의 브랜드/체급 점수, 별도 연결 시 사용
- `pricePercentile`: 같은 비교군 안에서 호텔 대비 가격 위치, 별도 연결 시 사용

구현 위치:

- `src/lib/scoring/hotel-quality.ts`
- `src/lib/scoring/recommendation-display.ts`
- 어드민 감사: `/api/admin/scoring/audit`, `/admin/scoring`

## 3. 판정 정책

- 호텔 등급 하나만으로는 `호텔 우수`를 주지 않는다.
- 4.5성 이상이라도 외부 품질/매칭/가격 근거가 없으면 기본적으로 `호텔 무난`이다.
- 외부 호텔 데이터가 좋고 매칭 신뢰도가 충분하면 `호텔 우수`로 올린다.
- 매칭 신뢰도가 낮거나 등급 정보가 부족하면 `호텔 확인 필요`로 낮춘다.

이 정책은 과장 방지용이다. 고객 신뢰를 위해 “좋아 보임”보다 “근거 있음”을 우선한다.

## 4. 운영자가 매일 볼 지표

`/admin/scoring`의 추천 데이터 감사에서 확인한다.

- 점수 보유 상품 비율
- 비교군 형성 비율
- 호텔 등급 누락률
- MRT 호텔 매칭률
- 쇼핑 횟수 누락률
- 옵션 수 누락률
- 직항 여부 누락률

우선순위:

1. 점수 보유율 80% 미만이면 재계산/등록 파이프라인부터 확인한다.
2. 비교군 형성률 50% 미만이면 목적지·출발일 정규화가 먼저다.
3. 호텔 등급 누락률 30% 초과면 호텔 추출/alias 보강이 먼저다.
4. 호텔 매칭률 30% 미만이면 MRT 매칭 큐를 보강한다.

## 5. 다음 개발 단계

- `hotel_aliases`, `package_hotels`, `hotel_score_snapshots`, `unmatched_hotels` 테이블로 호텔 마스터를 분리한다.
- 호텔명 자동 정규화 후 사람이 한 번 승인하면 이후 같은 호텔명은 자동 매칭한다.
- 예약/상담 결과로 “고객이 실제로 호텔 차이를 중요하게 봤는지”를 학습한다.
- 고객 화면은 점수 대신 “10만원 차이지만 호텔 조건이 더 좋아요” 같은 비교 문장으로 확장한다.
