## External Basis Checked

- Google Search Central says helpful content should be made primarily for people, not only to gain rankings, and asks teams to check whether automation creates substantial user value: https://developers.google.com/search/docs/fundamentals/creating-helpful-content
- Google spam policies classify large-scale automated content with little user value as scaled content abuse, regardless of whether it is generated by AI or other tooling: https://developers.google.com/search/docs/essentials/spam-policies
- Google recrawl guidance says reindexing can be requested after meaningful page changes, but crawling/indexing is not guaranteed and high-quality useful content is prioritized: https://developers.google.com/search/docs/advanced/crawling/ask-google-to-recrawl
- Search Console URL Inspection guidance says sitemap submission with updated `lastmod` is the better path for many updated URLs: https://support.google.com/webmasters/answer/9012289
- Google product snippet guidance requires product/offer/review-style structured data when a page is product-commercial enough to be eligible for product-rich results: https://developers.google.com/search/docs/appearance/structured-data/product-snippet
- Google Analytics recommended events define the commerce funnel events we should mirror in our own internal funnel: `view_item_list`, `select_item`, `generate_lead`, and `purchase`: https://support.google.com/analytics/answer/9267735

### Critical Gaps Found In The Initial Audit

- Informational posts do not use `post.destination` as the primary fallback for product recommendations. Several recommendation calls still depend on `pkg?.destination`, which is empty for information-only posts.
- Blog product cards use simple same-destination/price sorting, not `recommendBestPackages()`, `package_scores`, policy rank, or personalized scoring.
- `recommendation_outcomes.source` does not include `blog`, so blog recommendations cannot become first-class LTR/funnel training data.
- Blog product impressions are not recorded. We only know some users clicked a CTA, not which packages were shown, at what rank, or under what article intent.
- Blog product clicks do not capture `package_id`, rank, policy version, or intent. `BlogTracker` records only a boolean `cta_clicked`.
- `/api/packages/inquiry` increments inquiry counts but does not update `recommendation_outcomes` or `recommendation_events`.
- `blog-daily-summary` reports yesterday's publishing count but does not fail/alert when fewer than 3 posts were published.
- `blog-learn` collects weekly metrics and prompt suggestions, but does not yet consume `intent_quality` failures or blog recommendation funnel outcomes as a closed learning loop.

## Correct Direction

The correct architecture is not "publish more SEO pages." It is:

```text
Search intent
  -> helpful blog article
  -> article-intent-aware product recommendation
  -> product impression/click/inquiry/booking tracking
  -> recommendation_outcomes + content_attribution_events
  -> blog-learn and scoring policy feedback
  -> better next article and better next product card
```

## Target Design For Yeosonam OS

### 1. Blog Commercial Intent Layer

Extend `classifyBlogIntent()` output into a commercial intent profile:

- weather: season-safe, parent-friendly, low-weather-risk products
- preparation: low-friction, clear-inclusion, departure-ready products
- itinerary: matching duration and route-density products
- cost/currency: value and transparent-price products
- luxury/filial/activity: segment-specific products

The article stays helpful first. Product blocks should be contextual, not hard-sell.

### 2. Scored Blog Recommendation Section

Replace price-only curation in `src/app/blog/[slug]/page.tsx` with a shared helper:

```text
resolveBlogRecommendationContext(post)
  -> destination = post.destination || pkg.destination
  -> intent = classifyBlogIntent(...)
  -> candidates = recommendBestPackages(...) or package_scores
  -> display cards = best fit / easiest / premium or parent-friendly
```

Fallback is allowed only when scoring data is missing. Fallback must be logged as `recommendation_fallback=true`.

### 3. First-Class Blog Recommendation Tracking

Add `blog` to the recommendation source taxonomy and record:

- impression: article id, package id, rank, intent, policy id, session id
- click: same fields plus clicked package id
- inquiry: package lead/inquiry from same session
- booking: booking value from same session or content attribution

This aligns our internal funnel with GA4-style commerce events while preserving the existing Supabase analytics tables.

### 4. Daily SLA And Repair Loop

Daily publishing is not complete just because the cron ran. The daily standard should be:

- published count >= 3
- all new posts pass render/image/visual/SEO/editorial gates
- no more than 1 quality-gate failure without auto-repair
- if below 3, queue a replacement topic and create an admin/system alert

### 5. Existing Post Remediation And Reindexing

Existing posts should be updated, not blindly deleted or rewritten.

Priority score:

```text
GSC impressions or existing rank potential
+ editorial audit failure severity
+ active sellable products for the destination
+ blog engagement or CTA potential
+ seasonal freshness risk
```

Batch size: 10-20 posts at a time.

After each batch:

- keep slug and canonical if the URL has any value
- update body, metadata, product recommendation block, `updated_at`, sitemap `lastmod`
- run blog render/image/visual/SEO/editorial/revenue-funnel audits
- call bulk reindex only after gates pass
