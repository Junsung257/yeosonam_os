# 미매칭 큐에서 뽑은 상품등록/프로젝트 개선 인사이트

기준일: 2026-06-24

## 핵심 요약

이번 미매칭 큐 정리에서 가장 큰 발견은 "관광지가 부족해서 쌓인 문제"보다 "상품등록 입력을 너무 넓게 받아서 일정 문장 전체가 관광지 후보로 들어간 문제"가 더 컸다는 점이다.

운영 데이터 기준으로 active pending은 4,055건에서 254건으로 줄었고, 최종 `attraction_gap`는 0건이 됐다. 남은 254건은 관광지가 아니라 옵션투어 219건, 쇼핑 35건이다.

즉, 지금 프로젝트가 개발해야 할 방향은 관광지 DB를 무작정 많이 만드는 것이 아니라, 상품등록 단계에서 일정 문장을 정확히 구조화하고, 관광지/호텔/식사/이동/공지/옵션/쇼핑을 처음부터 분리하는 것이다.

## 발견된 주요 패턴

### 1. 관광지 큐가 실제로는 전체 일정 엔티티 큐가 되어 있었다

초기 active pending 4,055건에는 관광지뿐 아니라 호텔, 자유시간, 공지, 항공 안내, 가격 조각, 쇼핑, 옵션투어가 섞여 있었다.

개선 방향:

- 큐 이름과 대시보드 지표를 `관광지 미매칭` 하나로 보지 않는다.
- `등록가능 관광지`, `옵션/쇼핑 수동검토`, `공지/노이즈`, `호텔/비차단`, `후보 마스터`로 분리해서 본다.

### 2. 폭증은 특정 일자와 대량 업로드/재처리에서 발생했다

가장 큰 폭증은 2026-06-17 UTC였다. 이 날 내몽고, 다낭, 푸꾸옥, 시즈오카 계열 상품에서 일정 조각이 대량으로 들어왔다.

개선 방향:

- 대량 업로드 후 자동으로 `upload batch health`를 만든다.
- 특정 배치에서 미매칭이 급증하면 상품 공개 전에 차단하거나 경고한다.
- 배치별로 "관광지 후보 / 비관광 조각 / 수동검토" 비율을 저장한다.

### 3. 공급사 문장은 장소명이 아니라 설명문인 경우가 많다

예시:

- `아름다운 에메랄드 물빛과 고운 모래사장의 이국적인 캠비치`
- `산 전체가 대리석으로 만들어진 오행산 관광(마블마운틴)`
- `도심 속 자연과 낭만이 공존하는 계림 대표 산책 명소 용호공원`

이런 문장은 전체를 관광지명으로 만들면 안 되고, `캠비치`, `마블마운틴`, `용호공원`처럼 장소명만 추출해야 한다.

개선 방향:

- 상품등록 파서에 `장소명 추출기`를 별도 단계로 둔다.
- 설명문 전체는 attraction alias가 아니라 source evidence로 저장한다.
- 장소명 추출 실패 시 자동 등록하지 않고 후보 검토로 보낸다.

### 4. 상품명/공지/표 조각이 관광지로 오인되는 패턴이 반복됐다

예시:

- `카멜리아 후쿠오카 갓성비 시내핵심 3일`
- `PUS-FSZ`
- `총길이 430M`
- `※ 한국인 또는 한국어 가능 현지 가이드`
- `24~26일 출발)`
- `유류할증료`

개선 방향:

- 상품등록 전처리에서 표/공지/가격/항공코드/상품제목 조각을 먼저 제거한다.
- 같은 패턴이 2회 이상 나오면 fixture로 승격해서 회귀 테스트에 넣는다.
- 운영자가 무시 처리한 문구는 다음 배치부터 자동 무시 규칙으로 학습시킨다.

### 5. 옵션투어와 쇼핑은 관광지와 별도 도메인이다

최종 남은 254건은 옵션투어 219건, 쇼핑 35건이다. 이건 관광지 등록 문제가 아니라 판매/정책/노출 방식 문제다.

개선 방향:

- `optional_tours` 또는 `commercial_itinerary_items` 별도 관리 화면이 필요하다.
- 쇼핑 방문은 관광지 카드로 렌더하면 안 된다.
- 옵션투어는 가격, 포함/불포함, 선택 여부, 노출 정책을 별도 구조로 저장해야 한다.

### 6. 기존 관광지 alias 연결이 실제로 효과가 있었다

최종 파이프라인에서 기존 관광지에 alias를 붙여 해결된 건이 있었다. 즉 DB가 부족한 것만은 아니고, 같은 장소를 공급사마다 다르게 쓰는 문제가 크다.

개선 방향:

- 장소마다 `공식명`, `공급사 표현`, `한글 별칭`, `영문명`, `현지명`을 분리 저장한다.
- 자동 alias 추가는 고신뢰 매칭에서만 허용한다.
- 사람이 승인한 alias는 다음 상품등록부터 즉시 재사용한다.

### 7. 크론이 실패하면 프로세스가 있어도 없는 것과 같다

기존에는 orchestrator가 내부 HTTP 호출에서 401을 받아 첫 단계에서 멈췄다. 그래서 프로세스가 코드상 존재해도 운영상으로는 실행되지 않았다.

개선 방향:

- 크론 성공 여부를 기능별로 대시보드에 노출한다.
- `401`, timeout, active pending 증가를 운영 알림으로 만든다.
- 내부 단계는 가능하면 HTTP self-call이 아니라 서버 내부 함수 호출로 구성한다.

## 상품등록 개선 백로그

### P0. 신규 업로드 입구 차단

목표: 새 상품등록 때 같은 미매칭 큐가 다시 쌓이지 않게 막는다.

해야 할 일:

- 호텔/식사/이동/공지/가격/자유시간은 `unmatched_activities`에 attraction 후보로 넣지 않기
- 관광지 후보는 장소명 추출 성공 또는 기존 관광지 매칭 후보가 있을 때만 넣기
- 대량 업로드 후 `attraction_gap`, `manual_review`, `noise` 비율 저장

성공 기준:

- 신규 업로드 1건당 관광지 미매칭이 일정 조각 전체로 폭증하지 않는다.
- `attraction_gap`가 증가하면 어떤 상품/어떤 일자/어떤 문구 때문인지 바로 보인다.

### P1. 상품등록 리뷰 화면 개선

목표: 운영자가 "뭘 등록해야 하는지" 한 화면에서 판단한다.

해야 할 일:

- 관광지 후보, 옵션투어, 쇼핑, 공지/노이즈를 탭으로 분리
- 같은 문구 반복 건을 묶어서 한 번에 처리
- 승인/무시/alias 연결/후보 생성 결정을 다음 배치 규칙으로 저장

성공 기준:

- 운영자가 1,000개 행을 하나씩 보지 않는다.
- 반복 문구는 한 번 처리하면 같은 패턴이 다시 쌓이지 않는다.

### P1. 후보 마스터 검증 강화

목표: 고객 공개 관광지 자동 생성을 막고, 내부 후보만 안전하게 만든다.

해야 할 일:

- 외부 ID 또는 근거 출처가 있는 경우만 publishable 후보로 올리기
- Naver/Wikidata/OSM/공식 사이트/공급사 증거를 source evidence로 저장
- 설명문 전체를 이름으로 쓰지 않게 canonical name gate 강화

성공 기준:

- LLM 단독 판단으로 고객 공개 관광지가 생성되지 않는다.
- 내부 후보는 생성되더라도 `customer_publishable=false`로 시작한다.

### P2. 옵션투어/쇼핑 도메인 분리

목표: 관광지가 아닌 수익/정책성 항목을 별도 제품 기능으로 다룬다.

해야 할 일:

- 옵션투어 테이블 또는 구조화 필드 설계
- 쇼핑 방문 정책 필드 설계
- 고객 화면에서 옵션/쇼핑이 관광지 카드로 보이지 않게 렌더링 분리

성공 기준:

- `KISS OF THE SEA SHOW`, `면세점 1회`, `베트남 특산품 관광 3회` 같은 항목이 관광지 큐에 남지 않는다.

### P2. 운영 알림

목표: "개선했는데 안 돌았다"를 다시 만들지 않는다.

해야 할 일:

- cron 실패, timeout, 401 감지
- active pending 급증 감지
- 특정 업로드 batch에서 noise 비율 급증 감지

성공 기준:

- 크론 실패가 하루 이상 방치되지 않는다.
- 큐가 1,000개 이상 쌓이기 전에 알림이 온다.

## 전체 프로젝트 관점의 개발 기회

### 1. 상품등록을 단순 OCR/파싱이 아니라 데이터 품질 엔진으로 바꿔야 한다

현재 병목은 "텍스트를 읽는 것"보다 "읽은 텍스트가 어떤 업무 의미인지 분리하는 것"이다.

프로젝트 방향:

- OCR/입력 → 구조화 → 엔티티 분류 → 기존 DB 매칭 → 후보 검증 → 사람 승인
- 각 단계마다 실패 이유와 근거를 저장
- 운영자가 처리한 결정을 다음 배치에 학습

### 2. 관리자 화면은 총량보다 분해 지표가 중요하다

`pending 4,055`만 보면 전부 위험해 보인다. 하지만 분해하면 실제 관광지 문제, 옵션투어 문제, 쇼핑 문제, 잡음 문제가 다르다.

프로젝트 방향:

- 모든 운영 큐에 `총량 + 원인분해 + 자동처리 가능분 + 수동검토분`을 기본 제공
- 대시보드에서 "오늘 사람이 봐야 할 것"과 "시스템이 고쳐야 할 것"을 분리

### 3. AI는 최종 등록자가 아니라 후보 랭커/설명자여야 한다

이번 데이터는 LLM이 장소처럼 보이는 문장을 실제 장소로 오인할 위험을 잘 보여준다.

프로젝트 방향:

- AI는 후보를 만들고 이유를 설명한다.
- DB 생성은 규칙, 외부 증거, 사람 승인, 공개 가능 상태를 통과해야 한다.
- 고객 공개 데이터는 source-backed evidence 없이는 만들지 않는다.

### 4. 반복 패턴 학습이 운영 효율의 핵심이다

같은 문구가 여러 상품에서 반복됐다. 이건 사람이 매번 볼 문제가 아니라 규칙과 fixture로 승격해야 한다.

프로젝트 방향:

- 운영자가 처리한 alias/ignore/create 결정을 학습 데이터로 저장
- 다음 배치에서 자동 적용
- 회귀 테스트로 같은 실수 방지

## 바로 이어서 만들면 좋은 기능

1. 상품등록 배치 건강도 리포트

   업로드 1건마다 관광지 후보, 옵션투어, 쇼핑, 노이즈, 자동해결, 수동검토 수를 보여준다.

2. 반복 문구 자동 학습 화면

   운영자가 `무시`, `옵션`, `쇼핑`, `기존 관광지 alias`, `신규 후보` 중 하나로 처리하면 다음부터 자동 적용한다.

3. 옵션투어/쇼핑 관리 화면

   남은 254건처럼 관광지가 아닌 항목을 별도 정책 큐로 관리한다.

4. 크론/큐 장애 알림

   401, timeout, active pending 급증을 알림으로 보낸다.

5. 고객 공개 안전 게이트

   내부 후보와 고객 공개 관광지를 명확히 분리하고, 공개 전에는 외부 증거 또는 관리자 승인을 요구한다.
