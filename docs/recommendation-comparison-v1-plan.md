# 여소남 추천/비교 V1 개발 실행서

> 목적: 랜드사 패키지 상품을 고객 조건별로 비교해 “덜 후회할 선택”을 제안하고, 이후 반자유/자체 상품 기획에 쓸 고객 의도 데이터를 쌓는다.

## 1. 전략 판단

여소남은 자유여행 OTA와 정면 경쟁하지 않는다. V1의 포지션은 “패키지 상품 쇼핑몰”도 아니고 “AI 자유여행 플래너”도 아니다.

> 같은 날짜의 랜드사 상품을 가격, 호텔, 쇼핑, 옵션, 항공, 출발확정, 고객 성향 기준으로 비교해서 가장 덜 후회할 상품을 골라주는 플랫폼.

이 방향이 맞는 이유:

- 대형 여행사는 자체 상품/브랜드/항공 블록이 강하지만, 고객에게 “왜 이 상품이 나한테 좋은지”를 충분히 설명하지 못하는 경우가 많다.
- 자유여행 플랫폼은 이미 경쟁이 강하다. 여소남은 패키지와 자유여행 사이의 “선별/안심/반자유” 영역을 먼저 잡는 편이 현실적이다.
- 개별 상품 리뷰는 상품이 다시 등록되면 리셋되기 쉽다. 리뷰보다 상품군 비교, 랜드사 신뢰도, 호텔/옵션/쇼핑 구조화가 플랫폼 자산이 된다.
- 고객 행동 데이터가 쌓이면 항공 블록, 반자유 상품, 자체 기획 상품을 감이 아니라 수요 기반으로 만들 수 있다.

## 2. V1 원칙

V1은 완벽한 추천 알고리즘이 아니다. 목표는 설명 가능한 비교 추천과 고객 의도 데이터 수집이다.

- 점수보다 이유를 먼저 보여준다.
- “AI 추천 97점”처럼 단정하지 않는다.
- 리뷰가 없을 때는 별점 대신 비교판정을 보여준다.
- 단독 비교군 상품은 “1위”로 말하지 않는다.
- 실제 좌석/마감 데이터가 없으면 긴박감을 만들지 않는다.
- 연령대 높은 패키지 고객을 고려해 선택지를 줄이고 글자/버튼을 크게 쓴다.
- 고객에게 많이 고르게 하지 말고, 여소남이 먼저 골라주고 고객은 확인하게 한다.

## 3. 고객 UX 톤

토스처럼 쉽고 친근하게 간다. 단, 여행상품은 고관여 구매라서 귀여움보다 신뢰가 우선이다.

좋은 톤:

- “5/30 다낭 상품 6개를 비교했어요 🔍”
- “가격만 보면 2등, 편안함까지 보면 1등이에요”
- “10만원 차이, 호텔에서 꽤 크게 느껴질 수 있어요 🏨”
- “쇼핑 없이 여행에 집중할 수 있어요 🧳”
- “가격은 좋아요 💸 다만 선택관광 비용은 꼭 확인해보세요.”

피할 톤:

- “AI 최적화 추천”
- “무조건 최고”
- “오늘만 특가” 또는 근거 없는 마감 표현
- “추천점수 97점” 단독 노출
- “알고리즘 기반 개인화 랭킹” 같은 내부 용어

## 4. V1 고객 화면

### 4-1. 상품 카드

카드에는 다섯 가지만 먼저 보인다.

- 어디
- 언제
- 얼마
- 왜 추천인지
- 무엇을 누르면 되는지

예시:

```text
가족여행에 편해요 ✨

다낭 5/30 출발 · 3박5일
노쇼핑 · 호텔 우수 · 직항

1,000,000원~
[자세히 보기]
```

리뷰가 없을 때:

```text
여소남 비교판정
같은 날짜 상품 6개를 비교했어요
```

단독 비교군일 때:

```text
조건 확인 완료
가격·호텔·쇼핑 조건을 확인했어요
```

### 4-2. LP 상단

```text
5/30 다낭, 이 상품이 편해 보여요 ✨

노쇼핑 · 호텔 우수 · 직항
같은 날짜 상품 6개를 비교했어요

1,000,000원~
[내 여행에 맞는지 확인하기]
```

### 4-3. 비교 섹션

고객이 직접 복잡한 비교표를 조작하게 하지 않는다. 가장 헷갈릴 저가 상품 1개와 추천 상품 1개를 자동 비교한다.

```text
비슷한 80만원 상품도 있어요

80만원 상품은 더 저렴하지만,
쇼핑 3회와 선택관광 비용이 있을 수 있어요.

편하게 가려면 이 상품,
가격이 가장 중요하면 80만원 상품도 볼 만해요.
```

CTA는 최대 2개:

- `이 조건으로 상담하기`
- `저렴한 상품도 보기`

### 4-4. 고객 의도 선택

처음 질문은 1개만 한다.

```text
어떤 여행을 찾고 계세요?

[부모님/가족] [최저가] [쇼핑 없는 상품] [상담원이 골라줘요]
```

추후 V1.5에서 `호텔 우선`, `자유시간`, `골프`, `출발확정`을 확장한다.

## 5. 추천/비교 알고리즘 V1

V1은 룰 기반 정규화 + 현재 TOPSIS 점수 + 고객 의도별 문구 변환으로 간다.

### 5-1. 비교군

1순위:

```text
destination + departure_date + duration
```

2순위 fallback:

```text
destination + departure_date ± 3일 + duration
```

3순위 fallback:

```text
destination + 같은 월 + duration
```

단, fallback 비교군은 고객에게 “같은 날짜”라고 말하지 않는다.

### 5-2. 점수 축

내부 계산:

- 체감가 경쟁력
- 호텔/숙박 품질
- 쇼핑/옵션 부담
- 항공/이동 편의
- 일정 피로도
- 자유시간
- 랜드사 신뢰도
- 출발확정/좌석 안정성

고객 노출:

- `가격 좋아요 💸`
- `호텔 좋아요 🏨`
- `쇼핑 없어요 🧳`
- `부모님/가족에게 편해요 ✨`
- `조건 확인이 필요해요 ⚠️`

### 5-3. 체감가

기본가만 보지 않는다.

```text
체감가 =
기본 상품가
+ 필수 현지비용
+ 예상 선택관광 비용
+ 가이드팁
+ 싱글차지
+ 쇼핑/옵션 리스크 비용
- 포함 혜택 가치
```

V1에서는 기존 `computeEffectivePrice()`를 활용한다. 부족한 비용 데이터는 “확인 필요”로 노출하고, 숫자를 과장하지 않는다.

### 5-4. 호텔

V1에서는 호텔 외부 API를 붙이지 않는다. 현재 코드의 `hotel_avg_grade`, `hotel_brands`, `mrt_package_hotel_intel`을 활용해 보수적으로 노출한다.

고객에게 숫자 점수는 숨긴다.

- `호텔 우수`
- `호텔 무난`
- `호텔 확인 필요`

V1.5에서 `hotels`, `hotel_aliases`, `package_hotels`, `hotel_score_snapshots`, `unmatched_hotels`를 추가한다.

## 8. 성공 지표

4주 운영 후 판단한다.

- 추천/비교 배지 노출 상품 클릭률
- 추천 이유 열람률
- 비교표 열람률
- 상담 바텀시트 오픈률
- 카카오/전화 클릭률
- “최저가 아닌 추천 상품” 상담률
- 상담 후 이탈 사유
- 예약 후 취소/불만 비율
- 상담원이 추천 이유를 설명하기 쉬운지

V1 성공 기준:

- 리뷰 없는 상품에서도 상담 전환이 유지 또는 상승
- 추천 이유 열람 후 상담 전환이 일반 클릭보다 높음
- 최저가가 아닌 상품도 명확한 이유가 있으면 상담으로 이어짐
- 운영자가 오추천 원인을 추적할 수 있음

- `src/lib/scoring/recommendation-display.ts`: 고객 표시 문구가 호텔 V1.5 판정을 사용하도록 연결한다.
- `docs/hotel-scoring-v1-5-runbook.md`: 호텔 점수 운영 기준과 다음 DB 확장 방향을 별도 런북으로 분리한다.

## 2026-06-01 상담 피드백 루프 추가

추천/비교 V1의 정확도는 최초 설계만으로 고정하지 않고 상담 결과로 보정한다.

- `/api/admin/scoring/feedback`: 최근 `recommendation_outcomes`를 조회하고 상담원 피드백을 `notes`와 `outcome`에 반영한다.
- `/admin/scoring/funnel`: 최근 추천 노출/클릭 row를 상담원이 `선택`, `거절`, `호텔`로 표시할 수 있다.
- `선택`은 `outcome=inquiry`, `거절`은 `outcome=cancelled`로 반영해 `v_ltr_signals` 학습 라벨에 가까워지게 한다.
- 호텔 재확인은 outcome을 조작하지 않고 notes에만 남겨 호텔 alias/매칭 보강 큐로 활용한다.

이 단계의 목적은 “완벽한 추천 알고리즘”이 아니라, 상담원이 보는 실제 고객 반응을 다음 추천 가중치와 호텔 매칭 보강에 연결하는 폐쇄 루프를 만드는 것이다.
